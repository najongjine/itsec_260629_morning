p97
*** 응집도 : 높으면 좋은거
- 기능적응집도(functional) : 하나의 문제를 해결하기위해 모임. 목적이 같음.
- 순차적응집도(Sequential) : A모듈의 결과가 B모듈의 입력으로 들어감.
- 통신적응집도(communication) : A모듈의 결과를 B,C,D... 모듈들이 갔다씀.
- 절차적응집도(Procedural): 모듈들이 순차적으로 실행
- 시간적응집도(Temporal): 모듈들이 연관성은 없는데, 그냥 특정 시기에 수행됨
- 논리적응집도(logical): 비슷한 종류의 기능들을 한 모듈에 모아놨지만, 실제로는 서로 다른 일을 하는 상태
- 우연적응집도(Coincidential) : 제일꾸짐. 우연히 모듈들이 그냥 있음.

p99
*** 결합도 : 낮아야 좋은거
- 자료결합도(Data): A모듈에서 B모듈로 서로 소통할때 parameter 로 줘라.
  데이터를줄때 상수값 복사본을 줘서 원본에 손상이 안가는거
  int a=1; myfunc(a);
- Stamp결합도: A모듈에서 B모듈로 서로 소통할때 parameter 로 줘라.
  데이터를 줄때 주소로 줘라. (java, python에선 숫자,boolean,문자열 빼고는 전부다 stamp 결합도)
- 제어결합도(Control): 전달 대상에게 값만 전달하는게 아니라 제어요소도 함께 전달
  myfunc(int data, int type){
   if(type==1) 학생작업
   if(type==2) 교사작업
   else 행정작업
  }
- 외부결합도(External): 외부에 있는 데이터, 파일, 장치, 통신 프로토콜 등에 의존
   int count = 10; // 전역 변수

   void moduleA() {
      count++;
   }

   void moduleB() {
      printf("%d", count);
   }
- 공유결합도(common): 여러 모듈이 같은 전역 데이터 / 공통 데이터 영역을 같이 사용
   class GlobalData {
      public static int total = 0;
   }

   class A {
      void add() {
         GlobalData.total += 10;
      }
   }

   class B {
      void print() {
         System.out.println(GlobalData.total);
      }
   }
- 내용결합도(Content): 한 모듈이 다른 모듈의 내부를 직접 건드림
   class Account {
      public int balance = 10000;
   }

   class PaymentService {
      void pay(Account account) {
         // Account의 내부 데이터 balance를 직접 건드림
         account.balance -= 3000;
      }
   }

* API(Application Programming Interface)

        옜날에 web에서 message 교환 프로토콜로 쓰였다.

* UDDI : WSDL에 등록하면, 공개가 되었다.

* pipe filter pattern
- 데이터가 파이프를 타고 이동하면서, 중간중간 필터들이 데이터를 단계별로 처리하는 구조
   사용자가 입력한 글
   ↓
   욕설 제거
   ↓
   공백 정리
   ↓
   금지어 검사
   ↓

* Broker pattern
- 분산 시스템에서 요청을 중재하고 연결해주는 중개자
   클라이언트: 계산 서비스 필요해
   브로커: 알겠어. 적절한 서버 찾아서 연결해줄게
   서버: 계산 결과 줄게
   브로커: 클라이언트에게 결과 전달

* Event Bus pattern
- 여러 객체나 컴포넌트가 직접 서로 부르지 않고, 중간의 “이벤트 버스”를 통해 소식을 주고받는 구조
- subscripe(구독) & publish(발행)
   주문 코드: "주문완료 이벤트 발생!"
         ↓
      이벤트 버스
         ↓
   ├─ 재고 시스템: 재고 차감
   ├─ 문자 시스템: 문자 발송
   ├─ 이메일 시스템: 이메일 발송
   ├─ 배송 시스템: 배송 준비
   └─ 포인트 시스템: 포인트 적립

   주문 코드는 그냥 이벤트만 던집니다.

   "주문 완료됨"

   그 이벤트에 관심 있는 애들이 알아서 반응합니다.

p108
* MVC pattern
 Model: Data
 View : 화면, UI
 Controller: 서버의 api 주소. 로직담당

* Black Board pattern
- 여러 전문가 객체들이 하나의 공용 작업판을 보면서, 각자 아는 부분을 해결하고, 그 결과를 다시 작업판에 적어서 최종 답을 만들어가는 구조
   칠판 하나가 있다.

   문제를 풀어야 하는데, 한 사람이 다 못 푼다.

   그래서 여러 전문가가 온다.

   [공용 칠판]
   현재 문제 상황, 중간 결과, 힌트들이 적혀 있음

   전문가 A: 숫자 계산 담당
   전문가 B: 문장 해석 담당
   전문가 C: 그림 분석 담당
   전문가 D: 최종 판단 담당

   각 전문가는 칠판을 보고 말한다.

   전문가 A: "이 부분은 내가 계산할 수 있네."
   → 계산 결과를 칠판에 적음

   전문가 B: "A가 적은 결과를 보니 이 문장은 이런 뜻이네."
   → 해석 결과를 칠판에 적음


   전문가 D: "이제 정보가 충분하네. 최종 답은 이거다."

   이런 식으로 공용 저장소를 중심으로 여러 모듈이 협력하는 패턴이 블랙보드 패턴이다.


   여러 지식 소스 / 전문가 모듈
        ↓   ↑
    [ 블랙보드 ]
        ↓   ↑
   제어 컴포넌트

   블랙보드: 공용 데이터 저장소다. 문제 상태, 중간 결과, 최종 결과가 저장된다.
   지식 소스 Knowledge Source: 각각의 전문 처리 모듈이다. 자기가 해결할 수 있는 조건이 되면 블랙보드를 읽고, 처리 결과를 다시 블랙보드에 쓴다.
   제어 컴포넌트 Control Component: 어떤 지식 소스를 언제 실행할지 조정한다.

* 미들웨어
- A와B를 이어주는놈이라는 개념
- 예를들면: 
   DBMS(MySQL, postgres, oracle), 
   RPC(Remote Procedure Call): 원격에 있는 A컴퓨터에 있는 기능을 마치 로컬(내 컴퓨터)에서 내가가진것마냥 사용할수 있다.
   MOM(Message Oriented Middleware): 메시지 기반의 비동기형 메시지를 전달하는 방식
   TP-Monitor(Transaction Processing Monitor): 온라인 트랜젝션(업무)를 처리하고 감시,제어하는 미들웨어
   WAS(Web Application Server): 그냥 일반 서버라고 생각해도 됨. 데이터 엑세스, 세션관리, 트랜젝션(업무)관리, DB 연동.

요류대응 방안
* 이중화 : 서버 2대로 돌리기
* 재해 복구 시스템(DRS. Disaster Recovery System): 백업본 만들어놓기
  Mirror: 완전 동일한놈 원본이랑 같이 돌리기
  Hot site: 완전 동일한놈 stand by로 대기시키기
  Warm site: 중요성 높은 정보만 보관해둠
  Cold site: 최소한 정보만 보관해둠
* 복구목표시간(RTO, Recovery Time Objective): 서비스 중단때, 복구하기 위해 허용되는 최대 시간
* 복구목표시점(RPO, Recovery Point Objective): 장애가 났을 때 “최대 몇 시간/몇 분 전 데이터까지 날아가도 감당할 수 있냐”
